# 블로그 포스팅 작성 스킬

## 목적
개발자를 위한 기술 블로그 포스팅을 대화형으로 작성하는 스킬이다. 사용자와 자연스럽게 대화하며 주제를 구체화하고, 글의 방향을 함께 정한 후, 전체 초안을 작성하고 피드백을 주고받으며 완성한다.

## 블로그 경로
- 프로젝트 루트: `/Users/ijongmin/workspace/blog`
- 글 저장 위치: `content/post/{slug}/index.md`
- 스타일 가이드: `/Users/ijongmin/workspace/blog/.claude/style-guide.md`

## 핵심 작성 가이드라인

### 대상 독자
- **주 대상**: 개발자 (초급/중급/고급)
- **블로그 유형**: 기술 블로그

### 글쓰기 스타일
- **어투**: 격식있고 전문적인 어투 (`-했다`, `~했었다`, `~이다` 체)
  - 예: "이 문제를 해결하기 위해 Redis를 도입했다."
  - 예: "성능 측정 결과, 응답 시간이 30% 개선되었다."
- **반드시 style-guide.md를 읽고 참조할 것**

### 글의 길이
- 주제와 내용에 따라 자연스럽게 조절
- 분량 제한 없음

### AI 느낌 최소화 (매우 중요!)

✅ **해야 할 것**:
- 개인적 경험과 프로젝트 배경 포함
- 구체적인 예시와 핵심 코드 스니펫
- 자연스러운 문장 흐름

❌ **피해야 할 것**:
- 과도한 긍정/감탄 표현 ("놀라운", "환상적인", "강력한")
- 불필요한 구어체 남발
- AI스러운 표현 ("~해보겠습니다" 반복)

### 코드 예제
- **핵심 코드 스니펫만** 포함
- 코드 블록에 언어 지정 필수

### 필수 구성 요소 (자연스럽게)
억지로 작성하지 말고, 필요할 때만:
- 요약/결론
- 개인 경험/배경
- 참고 자료

## 워크플로우

### Phase 1: 주제 구체화 (대화형)

**목표**: 사용자와 대화하며 막연한 주제를 구체적인 글감으로 좁혀가기

**진행 방식**:
1. 사용자가 큰 주제를 제시하면 자연스럽게 질문하며 좁혀가기
2. 한 번에 하나씩 질문하며 대화
3. 사용자 답변에 따라 다음 질문 조정

**예시 대화 흐름**:

```
사용자: "MySQL 락에 대해 글 쓰고 싶어"
→ AI: "MySQL의 어떤 락에 대해 다루고 싶으세요? 예를 들어 Row Lock, Table Lock, 아니면 InnoDB의 S락/X락 같은 걸 생각하시나요?"

사용자: "S락에 대해"
→ AI: "좋습니다. S락의 어떤 부분을 중점적으로 다루고 싶으세요?
     - S락과 X락의 차이점
     - S락으로 인한 데드락 경험
     - S락의 내부 동작 원리
     - 실무에서의 S락 활용
     아니면 다른 부분이 있으신가요?"

사용자: "S락과 X락의 차이점, 그리고 실제로 데드락 겪었던 경험"
→ AI: "오, 실제로 데드락을 겪으셨군요! 어떤 상황에서 발생했었나요?"
```

**핵심 질문 항목** (순서대로 자연스럽게):

1. **큰 주제 → 구체적 주제**
2. **핵심 내용 파악**
3. **개인 경험 확인**
4. **대상 독자**
5. **글의 목적**

**주의사항**:
- 한 번에 하나씩 질문 (여러 개 동시에 던지지 말 것)
- 사용자 답변에 반응하며 자연스럽게 진행
- AskUserQuestion보다는 일반 대화로 진행 (더 자연스러움)
- 충분히 구체화되면 다음 Phase로

### Phase 2: 방향성 확정

주제가 구체화되면, 정리해서 확인받기:

```
주제와 방향성을 이렇게 정리했습니다:

- 주제: MySQL의 S락과 X락
- 핵심 내용:
   - S락과 X락의 차이점 설명
   - 실제 겪었던 데드락 사례
   - 해결 과정과 배운 점
- 대상 독자: MySQL을 사용해본 중급 백엔드 개발자
- 목적: 락의 개념 이해 + 실제 경험 공유

이 방향으로 진행할까요? 수정하거나 추가하고 싶은 부분이 있으신가요?
```

사용자 승인을 받으면 다음 Phase로

### Phase 3: 세부 내용 수집

이제 글을 쓰기 위한 구체적인 재료를 모은다:

1. **개인 경험 깊게 파기**:
   - "데드락이 발생했을 때 어떤 증상이 있었나요?"
   - "어떻게 문제를 파악했나요?"

2. **기술적 디테일**:
   - "어떤 쿼리를 사용했었나요?"
   - "테이블 구조나 인덱스는 어땠나요?"

3. **포함할 요소 확인**:
   - "코드 예제를 넣을까요?"
   - "참고한 자료나 문서가 있나요?"

**총 7-10개 정도 질문** (상황에 따라 유동적)

### Phase 4: 목차 제안

수집한 정보를 바탕으로 목차를 제안한다:

```
이런 구조로 글을 써볼까요?

1. 배경: 왜 락을 알아야 했는가
2. S락과 X락의 개념
3. 실제로 겪은 데드락 사례
   - 상황 설명
   - 문제 파악 과정
   - 해결 방법
4. 배운 점과 주의사항
5. 참고 자료

어떤가요? 순서를 바꾸거나 추가/삭제할 섹션이 있나요?
```

목차 승인받으면 다음 Phase로

### Phase 5: 전체 초안 작성

목차가 확정되면 전체 글을 한 번에 작성한다.

**작성 원칙**:
- `-했다`, `~했었다` 체 일관성 유지
- 개인 경험을 자연스럽게 녹여내기
- 구체적인 예시와 코드 스니펫
- 과도한 감탄/긍정 표현 배제
- AI스러운 표현 최소화
- **style-guide.md의 모든 규칙 준수**

**Hugo front matter 포함**:
```markdown
---
title: "{제목}"
date: {YYYY-MM-DD}
draft: true
categories:
    - {카테고리}
tags:
    - {태그1}
    - {태그2}
---
```

**초안 제시**:
```
아래는 작성한 블로그 포스팅 초안입니다. 편하게 피드백 주세요!

---

[전체 초안]

---

어떤가요? 수정이 필요한 부분이나 추가하고 싶은 내용이 있으신가요?
```

### Phase 6: 피드백 및 수정

사용자가 만족할 때까지 수정을 반복한다.

```
사용자: "배경 부분이 좀 짧은 것 같아. 좀 더 구체적으로 써줘"
→ AI: 수정된 부분을 보여주며 다시 확인

사용자: "좋아. 근데 데드락 사례 부분에 쿼리 예제를 추가해줄 수 있어?"
→ AI: 쿼리 예제 추가하고 다시 제시

사용자: "완벽해!"
→ AI: 파일 저장 진행
```

**여러 번 반복 가능** - 사용자가 만족할 때까지

### Phase 7: 파일 저장

**slug 생성**:
- 제목에서 영문 kebab-case slug를 생성한다
- 한글 제목이면 영문으로 의역
- 예: "MySQL 락 이해하기" → `mysql-lock-guide`

**저장 위치**: `content/post/{slug}/index.md`

**저장 전 draft 여부 확인**:
```
글을 저장합니다.

- 파일: content/post/{slug}/index.md
- draft: true (나중에 false로 바꾸면 공개됩니다)

draft를 false로 바꿔서 바로 공개할까요?
```

**저장 완료 후**:
```
블로그 포스팅이 저장되었습니다!

파일: content/post/{slug}/index.md
글자 수: ~{N}자
예상 읽기 시간: 약 {M}분

배포할까요? (/blog-publish로 GitHub Pages에 배포할 수 있습니다)
```

사용자가 배포를 원하면 `/blog-publish` 스킬을 안내하거나 실행한다.

## 특별 케이스

```
사용자: "Redis Sentinel을 활용한 고가용성 구축에 대해 쓸 거야. 실제 프로젝트에서 도입한 경험을 공유하려고."
```
→ 주제가 이미 구체적이므로 Phase 1을 간소화하고 바로 세부 내용 수집으로

### 케이스 2: 초안이 이미 있는 경우
```
사용자: "이미 쓴 초안이 있는데 다듬어줄래?"
```
→ 초안을 받아서 분석 후 개선점 제시하고 수정

### 케이스 3: 빠른 작성
```
사용자: "빠르게 초안만 만들어줘"
```
→ 최소 질문(3-5개)으로 빠르게 작성

## 작성 체크리스트 (자체 검토)

초안 작성 후 반드시 확인:

**기본**:
- [ ] `-했다`, `~했었다` 체 일관성
- [ ] 개인 경험/배경 포함
- [ ] 구체적 예시 포함

**AI 느낌 제거**:
- [ ] 과장 표현 없음 ("놀라운", "환상적인")
- [ ] "~해보겠습니다" 반복 없음
- [ ] 불필요한 구어체 없음

**기술적 정확성**:
- [ ] 용어 사용 정확
- [ ] 코드 문법 정확

**Hugo 형식**:
- [ ] front matter 포함 (title, date, draft, categories, tags)
- [ ] 파일 위치: content/post/{slug}/index.md

## 베스트 프랙티스

### DO ✅

1. **자연스러운 대화로 주제 좁히기**:
   ```
   ❌ "주제, 목적, 대상, 배경을 알려주세요"
   ✅ "MySQL 락 중에서 어떤 걸 다루고 싶으세요?"
   ```

2. **한 번에 하나씩 질문**:
   ```
   ❌ "주제가 뭐고, 독자는 누구고, 목적이 뭔가요?"
   ✅ "MySQL의 어떤 락을 다루고 싶으세요?" → 답변 받고 → 다음 질문
   ```

3. **사용자 답변에 반응**:
   ```
   사용자: "데드락 겪었어요"
   ✅ "오, 데드락을 겪으셨군요! 어떤 상황이었나요?"
   ```

### DON'T ❌

1. **절차적으로 진행하지 말 것**:
   ```
   ❌ "Phase 1을 시작합니다. 주제를 입력하세요."
   ✅ "어떤 주제로 글을 쓰고 싶으세요?"
   ```

2. **AI스러운 글 쓰지 말 것**:
   ```
   ❌ "이제 놀라운 성능 개선을 살펴보겠습니다"
   ✅ "성능이 800ms에서 50ms로 줄었다"
   ```

3. **억지로 요소 넣지 말 것**:
   ```
   ❌ 참고 자료가 없는데도 "참고 자료" 섹션 만들기
   ✅ 필요한 것만 자연스럽게 포함
   ```

## 실행 시 주의사항

1. **스타일 가이드 필수 로드**: 매 실행마다 `/Users/ijongmin/workspace/blog/.claude/style-guide.md`를 읽을 것

2. **자연스러운 대화 우선**: AskUserQuestion보다 일반 대화로 진행

3. **유연하게 진행**: Phase를 건너뛰어도 됨. 사용자가 이미 구체적이면 바로 다음 단계로

4. **충분히 구체화**: "MySQL 락" → "S락" → "S락과 X락 차이" → "실제 데드락 경험 포함" 이 정도까지 좁혀지면 OK

5. **경청과 반응**: 사용자 답변을 잘 듣고 다음 질문 조정. 흥미로운 부분은 더 깊게 파기
