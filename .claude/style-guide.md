# 블로그 포스팅 스타일 가이드

이 문서는 블로그 글의 문체, 톤, 용어 사용 등의 기준을 정리한 가이드입니다.
모든 블로그 관련 스킬(blog-write, blog-new 등)에서 참고합니다.

---

## 1. 어투 및 문체

### 1.1 기본 어투

**`-했다` / `~이다` / `~한다` 체** (해라체)를 사용한다.

```
✅ 올바른 예:
- "이 문제를 해결하기 위해 Redis를 도입했다."
- "성능 측정 결과, 응답 시간이 30% 개선되었다."
- "각 버전이 나온 배경과 바뀐 점을 살펴본다."
- "주의할 점은 인덱스 레코드에 락이 걸린다는 것이다."

❌ 잘못된 예:
- "이 문제를 해결하기 위해 Redis를 도입했습니다." (합쇼체 ❌)
- "살펴보겠습니다." (합쇼체 ❌)
- "살펴볼게요." (해요체 ❌)
- "도입해봤음." (해라체지만 구어체 ❌)
```

### 1.2 시제

- **설명/원리**: 현재형 (`~한다`, `~이다`)
- **경험/사례**: 과거형 (`~했다`, `~했었다`)
- **독자 행동 유도**: 현재형 (`~하면 된다`, `~할 수 있다`)

```
✅ "fetch는 원격 저장소의 변경사항을 로컬에 가져오기만 한다." (설명 → 현재형)
✅ "하지만 큰 문제가 있었다." (과거 사실 → 과거형)
✅ "위 명령은 다음과 같은 커밋들을 확인할 수 있다." (유도 → 현재형)
```

### 1.3 문장 길이와 호흡

- 한 문장은 **40~60자** 내외가 적당하다
- 너무 긴 문장은 두 문장으로 나눈다
- 짧은 문장과 긴 문장을 적절히 섞어 리듬감을 준다

### 1.4 일관성 규칙

- **한 글 안에서 어투를 섞지 않는다**: `~했다`와 `~했습니다`를 같이 쓰지 않는다
- **존댓말과 반말을 섞지 않는다**: 본문 전체에서 통일

## 2. AI스러운 표현 회피

### 2.1 금지 표현

다음 표현들은 AI가 생성한 글처럼 보이게 만드므로 사용하지 않는다.

**과도한 감탄/긍정**:
```
❌ "놀라운", "환상적인", "강력한", "혁신적인", "획기적인"
❌ "정말 대단한 기능이다"
✅ "유용한 기능이다", "실무에서 자주 쓰인다"
```

**AI 특유 패턴**:
```
❌ "~해보겠습니다" 반복 사용
❌ "자, 이제 ~를 살펴볼까요?"
❌ "그럼 하나씩 알아보겠습니다!"
❌ "먼저 ~에 대해 알아보겠습니다"
❌ "이번 섹션에서는 ~를 다루겠습니다"
✅ "~를 살펴본다", "~를 다룬다"
```

**과도한 연결사/접속사**:
```
❌ 모든 문단을 "또한", "더불어", "아울러"로 시작
❌ "뿐만 아니라", "나아가"를 과도하게 사용
✅ 자연스러운 문장 전환, 때로는 연결사 없이 바로 시작
```

**불필요한 수식어**:
```
❌ "매우 중요한", "반드시 알아야 할", "핵심적인"을 남발
✅ 정말 중요한 곳에서만 강조, 나머지는 담백하게
```

### 2.2 자연스러운 글쓰기 패턴

```
✅ 질문형 도입: "왜 이렇게 바뀌어야 했는가?"
✅ 간결한 진술: "이게 전부다."
✅ 대조/비교: "편리하지만 주의가 필요하다."
✅ 개인 경험 섞기: "실무에서 마주치는 상황"
✅ 직접적 표현: "큰 문제가 있었다."
```

## 3. 기술 용어 일관성

### 3.1 한글/영문 혼용 규칙

- **처음 등장 시**: 한글(영문) 형식으로 병기
- **이후**: 한글 또는 영문 중 하나로 통일 (해당 글 내에서)
- **코드/명령어**: 항상 영문 그대로 (백틱 사용)

### 3.2 자주 쓰는 기술 용어 통일

| 영문 | 권장 표기 | 비권장 |
|------|-----------|--------|
| commit | 커밋 | 코밋 |
| branch | 브랜치 | 브런치 |
| merge | merge 또는 병합 | 머지 |
| rebase | rebase | 리베이스 |
| fetch | fetch | 페치 |
| pull | pull | 풀 |
| push | push | 푸시 |
| conflict | 충돌 또는 conflict | 컨플릭트 |
| deploy | 배포 | 디플로이 |
| cache | 캐시 | 캐쉬 |
| thread | 스레드 | 쓰레드 |
| parameter | 파라미터 | 파라메터 |

### 3.3 약어 사용

- 첫 등장 시 풀네임 병기: "TCP(Transmission Control Protocol)"
- 이후 약어만 사용: "TCP"
- 개발자에게 익숙한 약어는 병기 생략 가능: HTTP, API, SQL, CSS 등

## 4. 구조와 포맷

### 4.1 코드 블록

- 언어 지정 필수 (```bash, ```sql, ```javascript 등)
- 핵심 코드만 포함 (불필요한 보일러플레이트 제거)
- 주석은 한국어로 달되, 과하지 않게

### 4.2 강조

- **볼드**: 핵심 키워드, 중요 개념
- `백틱`: 코드, 명령어, 파일명
- *이탤릭*: 거의 사용하지 않음

## 5. 맞춤법 및 띄어쓰기

### 5.1 자주 틀리는 맞춤법

| 올바른 표현 | 잘못된 표현 |
|-------------|-------------|
| 됐다 | 됬다 |
| ~할 수 있다 | ~할수있다 |
| 예를 들어 | 예를들어 |
| 안 되다 | 안되다 |
| ~할 때 | ~할때 |
| 웬만하면 | 왠만하면 |
| 금세 | 금새 |

### 5.2 띄어쓰기 규칙

```
✅ "할 수 있다" (보조 용언 띄어쓰기)
✅ "하고 있다"
✅ "할 때"

❌ "할수있다"
❌ "하고있다"
❌ "할때"
```

### 5.3 외래어 표기

- 개발자 커뮤니티에서 통용되는 표기 우선
- 예: "캐시" (O), "캐쉬" (X) / "스레드" (O), "쓰레드" (X)

## 6. 글의 흐름

### 6.1 도입부

- **왜 이 글을 읽어야 하는지** 짧게 설명
- 독자가 공감할 수 있는 문제 상황으로 시작
- 길게 쓰지 않는다 (3~5문장)

### 6.2 본문 전개

- 개념 설명 → 예시 → 실무 적용 순서
- 각 섹션이 자연스럽게 연결되도록
- "왜?"를 항상 설명한다

### 6.3 마무리

- 억지로 "요약"이나 "결론" 섹션을 넣지 않는다
- 자연스러운 마무리가 없으면 없는 대로 끝낸다
- 필요할 때만 참고 자료 링크를 추가한다
